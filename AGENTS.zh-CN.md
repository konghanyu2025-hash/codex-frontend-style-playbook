# 前端代理规则

当代理使用本仓库构建或改进前端时，遵循这些规则。

## 目标

构建真实、可用、经过截图验收的前端，而不是只完成表面样式。

不要依赖模糊风格词。先把“高级”“现代”“干净”“有创意”等词转成布局、密度、排版、颜色、组件、动效和响应式规则。

## 必需流程

1. 读取任务，识别产品类型、用户、主要流程和必需内容。
2. 按 `workflows/zh-CN/style-discovery.md` 完成风格发现。
3. 按 `workflows/zh-CN/gpt-image-2-anchor.md` 生成或收集视觉锚点。
4. 产出 `style_brief.md`、`visual_anchor.md`、`implementation_brief.md`。
5. 只有当 brief 足够具体时才开始实现。
6. 完成 `workflows/zh-CN/visual-qa.md` 后再交付。

## 前端标准

- 构建真实产品页面、应用、工具或站点。除非用户明确要求，不要默认做营销落地页。
- 优先使用现有项目的技术栈、组件系统、CSS 约定、图标、路由和数据模式。
- 使用真实的信息架构和用户流程。
- 首屏应该有用，而不只是装饰。
- 为面板、工具栏、卡片、网格和控件定义稳定尺寸，避免悬停或文本变化导致布局跳动。
- 移动端布局必须有意设计，而不是把桌面端硬挤进去。
- 确保按钮、卡片、导航、标签页和指标块中的文本不会溢出。
- 除非产品确实需要新手引导，不要在 UI 内写大量解释如何使用界面的说明文字。

## 视觉方向规则

- 使用视觉锚点提取规则，而不是复制像素。
- 把“高级”“现代”“干净”“活泼”“大胆”等词转成具体选择。
- 除非 brief 明确要求，避免整个界面只由同一色系的变化构成。
- 避免装饰性光斑、通用渐变圆球、空洞 Bento 网格和库存图感很强的首屏。
- 当网站需要视觉资产时，使用与产品相关的图像或媒体。
- SaaS 和业务工具应优先使用克制、紧凑、可扫描的界面，而不是夸张的营销页结构。
- 创意作品集或表达性网站可以更有视觉观点，但仍要保持导航和阅读清晰。

## 实现规则

- 除非项目已经使用或用户明确要求，不要引入新的 UI 库、CSS 框架或图标集。
- 只有当重复结构或真实复杂度需要时才添加抽象。
- 响应式布局应在结构设计中解决，而不是最后用补丁修。
- 为需要交互的元素提供空、加载、错误、选中、悬停、聚焦和禁用状态。
- 对真实操作使用可访问标签和键盘友好的控件。
- 把截图当作证据。如果截图和预期风格冲突，就修 UI。

## 最终验收

交付前确认：

- 桌面端视口完整且构图正确。
- 移动端没有文本裁切、控件隐藏或堆叠混乱。
- 主要操作清晰可见。
- 视觉层级符合产品目标。
- UI 不像未设计过的模板。
- 实现结果匹配 `implementation_brief.md`。
